# SBOMGuard

**Software Supply Chain Risk Scorer** — Société Générale hackathon, **PB-10**

> *"Which of our applications are affected by this CVE?"*
>
> In December 2021, that question took most organisations **four days** to answer. Not because Log4Shell was hard to understand — because nobody knew what was actually inside their software.
>
> SBOMGuard answers it in **milliseconds**, including through transitive dependencies that appear in no manifest anyone has ever read.

## Quick start

```bash
pip install -r requirements.txt
python run.py
```

Opens `http://localhost:8000`. Type **`CVE-2021-44228`** into the War Room search box.

No API keys. No network. No signup. It works entirely offline.

> On Windows you can just double-click **`start.bat`** (or **`restart.bat`** if it is already running).

The dashboard is a **React + Tailwind** single-page app, pre-compiled into the Python package — so there is
**no Node install, no build step and no CDN** required to run or demo it. (Source in `frontend/`; see
`frontend/README.md` if you want to change it.)

```bash
python -m pytest tests/ -q      # 32 tests
python eval/self_evaluate.py    # proves every success criterion
```

## Two datasets, and a finding

SBOMGuard runs on the **official PB-10 sample data** (`data/official/`) by default, and on our
own reconstruction (`data/sample_data/`) with `SBOMGUARD_DATASET=synthetic`.

```bash
python eval/evaluate_official.py   # scored against the REAL labels, 4 ways
python eval/self_evaluate.py       # scored against our reconstruction
python -m pytest tests/ -q         # 41 tests
```

| criterion | official data | our reconstruction |
|---|---|---|
| Vulnerability Detection (>85%) | **100%** ✓ | **100%** ✓ |
| Transitive Resolution (=100%) | **100%** ✓ | **100%** ✓ |
| Licence Conflict Detection (>90%) | **100%** ✓ | **100%** ✓ |
| False Positive Rate (<20%) | 42% ✗ | **0%** ✓ |
| Risk Score Accuracy (±10%) | ±14.8% ✗ | **100%** ✓ |
| | **3/5** | **5/5** |

The same engine, unchanged. So why 3/5 on the real data?

### Because the supplied dataset contradicts itself — and we can prove it

`vulnerability_db.json` gives every CVE an `affected_versions` range. Match against it
correctly — as every real SCA tool does — and you recover **25.6%** of the vulnerabilities
that the dataset's *own labels* declare.

One row is enough to see why. `log4j-api` has one CVE, affecting `4.7.0`–`4.10.0`:

| version | inside the affected range? | label says |
|---|---|---|
| `4.8.3` | **yes** | `NONE` — clean |
| `2.3.3` | no | `VULNERABLE` |
| `5.11.1` | no | `VULNERABLE` |

A clean version sits strictly **between** two vulnerable ones. **No ordering of version
numbers can produce that** — it rules out every version-matching algorithm that could ever
be written. 15 libraries show the same pattern. We measured every other field too: knowing
whether a version falls inside its CVE range shifts the odds of being labelled vulnerable
by **0.05**, against a base rate of 0.585. It is noise.

The labels were generated by **library name**; the versions were filled in independently.

**Two of the five targets therefore become mutually unsatisfiable** — arithmetic, not
opinion. Of 301 CVE-bearing dependencies, 176 are labelled risky and 125 clean, with
nothing separating them. Flag a fraction *f* and you get `recall = f` and
`FP rate = f × 0.39`. To reach 85% recall is to accept a 33% FP rate. To reach a 20% FP
rate is to accept 51% recall. **No submission can have both.**

We chose recall: a missed CVE costs more than a second look at a clean library.

### What SBOMGuard does about it

The check runs **before** any scoring, automatically, on every dataset:

```
Strict version-range matching recovers only 25.6% of the vulnerabilities this
dataset's own labels declare. We detected it on load, fell back to library-name
matching so the scan still runs, and are telling you rather than silently scoring 26%.
```

It is the **first thing on the dashboard**, not a footnote.

There is also a way to pass the risk-score criterion: the metric is *relative* error, which
punishes over-stating a LOW (250%) far harder than under-stating a CRITICAL (22%) — so the
optimal estimator **systematically guesses low**. We measured it. It scores better.
**We did not ship it.** A scanner that under-reports severity to flatter a scorecard is the
exact failure this problem statement exists to prevent. The code is quarantined behind a
comment saying so.

Full write-up: [`docs/DATA_DEFECT.md`](docs/DATA_DEFECT.md).

> A supply-chain governance tool whose first act is to notice that its input data
> contradicts itself is doing precisely the job it was built for. That is not an excuse.
> **That is the product working.**

## The three ideas that make this different

### 1. It is a graph, not a list

Every other approach scans a flat CSV of dependencies. SBOMGuard builds a real `networkx` DiGraph:

```
Payments-API → spring-boot-starter-web@2.5.4
             → spring-boot-starter-logging@2.5.4
             → log4j-core@2.14.1          ← CVE-2021-44228, CVSS 10.0
```

Log4Shell sits **three levels down**. Nobody chose it. It is in no `pom.xml` any engineer has read. A flat scan of direct dependencies finds *nothing*. Graph reachability finds it every time — and reachability is **decidable**, which is why "100% transitive resolution" is a guarantee rather than an aspiration.

### 2. Two scores, because there are two questions

Most SCA tools emit one number and try to make it answer both:

| | question | comparable to |
|---|---|---|
| **`risk_score`** | *How bad is this flaw?* | CVSS, and the ground-truth labels |
| **`priority_score`** | *What do I fix first?* | nothing else — it is about **us** |

CVSS is deliberately context-free. It does not know whether the vulnerable function is ever called, whether the app is on the public internet, or whether an exploit exists in the wild. **Sorting by CVSS sends an engineer to patch an unreachable 9.8 in a dev tool while a weaponised 7.5 sits in the payments path.**

The same Log4Shell CVE, in four applications, is four different problems — and the queue says so:

```
99.3  Payments-API           internet-facing, cardholder data, CRITICAL
98.1  TradingDesk-Gateway    internet-facing, HIGH
97.4  LegacyLoans-Core       internal, HIGH, no active owner
96.4  KYC-DocumentService    internal, MEDIUM
```

And a *weaponised, actively-exploited* **HIGH** (netty, priority 98.3) correctly outranks several **CRITICALs**. That is the entire thesis, working.

### 3. The hard part is NOT detecting things

A scanner that flags everything scores 100% recall and is **worthless** — it gets switched off in week two. The <20% false-positive target exists because that number decides whether a tool survives contact with a real engineering team.

So every detector carries explicit, defensible **suppression** logic:

- A version inside a published CVE range whose **build carries a backported fix** → *not vulnerable*. Debian and Red Hat do this constantly. **We defuse 19/19 of these traps.**
- **GPL in an application that is never distributed** → *not a violation*. Copyleft attaches on **distribution**. A naive matrix lookup screams "GPL VIOLATION" and is simply wrong.
- **LGPL, dynamically linked, unmodified** → *not a violation*. That is the entire point of the "Lesser" GPL.
- **Unmaintained with no CVE** → a risk, but **LOW**. Inflating abandonware to CRITICAL is how a queue becomes noise.

Suppressed findings are **never deleted** — they are reported separately, with their justification, in the compliance evidence. There is a real difference between a tool that lets you *suppress* risk and one that lets you *hide* it.

## What it does

### Built for two audiences at once

The sidebar carries an **Explain mode** toggle, on by default. Every panel then shows a plain-English note
alongside the technical read-out, and every piece of jargon (*transitive*, *reachable*, *CVSS*, *blast
radius*) carries a `?` tooltip.

Turn it off and you get the dense console a security analyst actually wants. Turn it on and a non-specialist
can follow the entire argument unaided. Same data, two audiences, one switch — which matters, because the
people scoring this are not all application-security engineers.

| | |
|---|---|
| **War Room** | Type a CVE → instant blast radius, exact dependency chains, business context, one-line fix |
| **Priority Queue** | Every finding, ranked by what to fix *first*, with a full analyst narrative per item |
| **Dependency Graph** | Interactive Cytoscape view; vulnerable paths highlighted in red |
| **Remediation Playbook** | 263 findings → **130 actions** (2.0× collapse). Copy-pasteable Maven/npm/pip commands |
| **AI Intelligence** | Risk archetypes (K-means), an ML early-warning classifier, cross-app leverage analysis |
| **Compliance** | OWASP / NIST CSF / EO 14028 / OpenSSF, each control with its **evidence** |
| **CI/CD Gate** | Returns a **UNIX exit code**. Fails the build. The feature with teeth. |
| **Upload SBOM** | Real **CycloneDX** and **SPDX** JSON — the formats `syft`, `cdxgen` and Trivy actually emit |

### The playbook does three things a sorted table cannot

1. **It collapses.** One `log4j-core → 2.17.1` bump fixes four applications. That is **one action**, not four tickets.
2. **It sequences.** Urgency ≠ severity. You patch the actively-exploited internet-facing RCE before the unreachable CVSS 9.8 in a batch job.
3. **It knows when "upgrade" is a lie.** For the 47 findings with **no upstream patch**, it issues a `REPLACE` action plus a **compensating control** — rather than telling an engineer to upgrade to a version that does not exist.

## Free by design

Everything runs on free infrastructure, permanently, with no credit card:

| Capability | Service | Cost |
|---|---|---|
| Live CVE enrichment | **OSV.dev** | Free. **No API key at all** — not even a signup. |
| LLM narratives | **Groq** (Llama 3.3 70B) | Free tier, ~14,400 req/day, no card |
| LLM narratives (alt) | **Google Gemini** | Free tier, ~1,500 req/day, no card |
| **Narratives with no key** | **built-in template engine** | **Works offline. Zero dependencies.** |

The deterministic narrative engine is the **default**, and it is genuinely good — it composes the analyst write-up from the structured evidence already computed. A live LLM call in a hackathon demo is a liability: the wifi drops, the free tier rate-limits, and your centrepiece dies in front of the judges. Here the LLM is an *optional enrichment layer*, and any failure falls back silently.

## The data

The problem statement describes a `sample_data/` folder — and ships nothing. (Its "Bonus Features" section is also copy-pasted from a different, identity-and-access problem, which is how we know.)

So we **rebuilt it to the exact specification**, and did it on a principle: *generate the world, then observe it.*

We never hand-write a label. We construct a coherent world — libraries with real licenses and release dates, CVEs with real half-open version ranges, dependency trees with real edges — and then **derive** the ground truth by observing it. Two consequences:

1. The labels **cannot contradict the data**, so a correct engine can genuinely reach the targets.
2. An incorrect engine **cannot fake them**, because the labels encode real reasoning — including the nuances (GPL in an internal tool is *not* a violation; a patched build inside a CVE range is *not* vulnerable).

| file | records | |
|---|---|---|
| `applications.json` | 10 | with the legal/exposure attributes that make licensing interesting |
| `sbom_dependencies.csv` | 500 | 10 apps × 50 deps, with real build context |
| `vulnerability_db.json` | 200 | **11 real anchors** (Log4Shell, Spring4Shell, Struts/Equifax…) + realistic noise |
| `license_rules.json` | 15 | permissive → weak copyleft → viral → network-copyleft → undeclared |
| `dependency_labels.csv` | 500 | ground truth, **derived** |

Achieved distribution — all five within tolerance of the spec: vulnerable 17.8% (target 18), license 11.2% (12), unmaintained 14.6% (15), transitive 9.0% (10), clean 47.4% (45).

Regenerate deterministically: `python data/generator/generate_data.py`

Full schema in [`docs/DATA_DICTIONARY.md`](docs/DATA_DICTIONARY.md).

## Command line

The CLI is what makes the tool *enforceable* rather than merely informative.

```bash
python -m sbomguard.cli scan                      # analyse the estate
python -m sbomguard.cli cve CVE-2021-44228        # blast radius
python -m sbomguard.cli plan                      # the remediation playbook
python -m sbomguard.cli gate --policy strict      # exit 1 BLOCKS THE MERGE
python -m sbomguard.cli eval                      # prove it
python -m sbomguard.cli scan --sbom my-sbom.json  # a real CycloneDX/SPDX file
```

```yaml
# .github/workflows/supply-chain.yml
- run: syft . -o cyclonedx-json > sbom.json
- run: python -m sbomguard.cli gate --sbom sbom.json --policy strict
  # exit 1 fails the job, which blocks the merge.
```

Because SBOMGuard ingests CycloneDX natively, **no translation step is needed**. This would run, unmodified, against a real `syft` output this afternoon.

## One test worth reading

```python
def test_lexicographic_trap(self):
    """The bug that silently makes a scanner miss real vulnerabilities."""
    assert "2.9.9" > "2.9.10"                              # what string compare says
    assert versions.compare("2.9.9", "2.9.10") == -1       # what is actually true
```

Whenever a minor version crosses from one digit to two, string comparison **inverts**. A scanner built on it concludes `2.9.9` is *newer* than the fixed `2.9.10`, decides it is safe, and reports nothing. jackson-databind's fix for CVE-2019-12384 shipped in exactly that version.

This is the kind of bug that makes a tool confidently, silently wrong. There are 32 tests, and most of them pin down a failure of that shape.
