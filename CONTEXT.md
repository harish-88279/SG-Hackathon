## 1. What this is

**SBOMGuard** — a submission for **Société Générale's GRC Hackathon, Problem Statement 10:
Software Supply Chain Risk Scorer (SBOM Analyzer)**.

The brief, in one sentence: *when a critical vulnerability like Log4Shell drops, which of our
applications are affected — including through dependencies nobody chose?*

The tool ingests SBOMs, resolves the full transitive dependency graph, cross-references a CVE
database, checks licence compatibility, flags unmaintained libraries, scores per-application
risk, and outputs a ranked remediation plan. There is a React dashboard, a CLI, a REST API,
and a CI gate.

**Status: complete and working.** 41 tests pass. It runs on both the official SG dataset and
our own reconstruction.

## 2. The single most important thing to know

**The official dataset supplied by SG contradicts itself, we proved it, and that proof is the
centrepiece of the submission.**

`vulnerability_db.json` gives each CVE an `affected_versions` range. Match against it correctly
— as every real scanner does — and you recover only **25.6%** of the vulnerabilities that the
dataset's *own labels* declare.

One row shows why. `log4j-api` has one CVE, affecting `4.7.0`–`4.10.0`:

| version in SBOM | inside the affected range? | their label says |
|---|---|---|
| `4.8.3` | **yes** | `NONE` (clean) |
| `2.3.3` | no | `VULNERABLE` |
| `5.11.1` | no | `VULNERABLE` |

A clean version sits strictly **between** two vulnerable ones. **No ordering of version numbers
can produce that** — it rules out every version-matching algorithm that could ever be written.
15 libraries show the same pattern. We measured every other input field too: knowing whether a
version falls inside its CVE's range shifts the odds of being labelled vulnerable by **0.05**,
against a base rate of **0.585**. It is noise.

**The labels were generated by library name; the versions were filled in independently and never
reconciled.**

### Consequence: two of the five success criteria are mathematically unreachable

Of 301 CVE-bearing dependencies, 176 are labelled risky and 125 clean, and *nothing in the
inputs separates them*. Flag a fraction *f* and you get `recall = f` and `FP rate = f × 0.39`.

- 85% recall → **33% false positives** (target <20%) ✗
- 20% false positives → **51% recall** (target >85%) ✗

**No submission can satisfy both.** We chose recall — a missed CVE costs more than a second look
at a clean library.

Full write-up with the proof: **`docs/DATA_DEFECT.md`**. Read it.

### What the tool does about it

A **data-quality control runs on load, before any scoring** (`official.diagnose_version_data()`).
If a correct range matcher recovers <50% of the declared vulnerabilities, the version data is not
describing the same world as the labels. The tool then:

1. falls back to library-name matching so the scan still runs,
2. surfaces the finding as an **amber banner on the front page of the dashboard**, not a footnote.

> A supply-chain governance tool whose first act is to notice that its input data contradicts
> itself is doing precisely the job it was built for. That is not an excuse. **That is the
> product working.**

### One thing we deliberately refused to do

The risk-score metric is **relative** error, `|ours − theirs| / theirs`. That loss is savagely
asymmetric: calling a LOW a HIGH costs **250%**, calling a CRITICAL a HIGH costs **22%**. So the
estimator that minimises it **systematically guesses low**. We implemented it, measured it, and it
scores better.

**We did not ship it.** The code exists at `scoring.representative(strategy="optimistic")`,
quarantined behind a comment explaining exactly why it must never be used. A scanner that
under-reports severity to flatter a scorecard is the precise failure this problem statement
exists to prevent.

**If you are asked to "improve the score", do not touch this.** It is a deliberate, documented,
principled refusal, and it is worth more than the point it would win.

## 4. How to run it

```bash
pip install -r requirements.txt
python run.py                      # → http://localhost:8000
```

On Windows just double-click **`start.bat`** (or **`restart.bat`** if it's already running —
that frees port 8000 first).

The dashboard is a **pre-compiled React bundle** in `src/sbomguard/static/ui/`. **You do not need
Node to run or demo this.** No CDN either — it renders from disk, deliberately, because a demo
that needs conference wifi is a demo that fails.

### Everything else

```bash
python eval/evaluate_official.py   # scored against the REAL SG labels, 4 ways + the defect proof
python eval/self_evaluate.py       # scored against our reconstruction (5/5)
python -m pytest tests/ -q         # 41 tests

python -m sbomguard.cli scan                 # analyse the estate
python -m sbomguard.cli cve CVE-2024-1060    # blast radius for one CVE
python -m sbomguard.cli plan                 # the remediation playbook
python -m sbomguard.cli gate --policy strict # CI gate — exit 1 BLOCKS the merge
```

### Datasets

Two of them. Default is **official**.

```bash
SBOMGUARD_DATASET=official    # data/official/   — the real SG files (default)
SBOMGUARD_DATASET=synthetic   # data/sample_data/ — our own reconstruction
```

The synthetic one exists because when this was built, SG's problem-statement webpage described a
`sample_data/` folder but shipped nothing (its "Bonus Features" section was even copy-pasted from
a different, identity-and-access problem). So we rebuilt the dataset to spec. The real files were
found later, in `GRC-Hackathon-main`, and are now vendored at `data/official/`.

Keep both. The synthetic set is the **control**: it proves the engine hits 5/5 on
internally-consistent data, which is what makes the official-data failures attributable to the
data rather than to us.

### Changing the UI (only if you must)

```bash
cd frontend
npm install
npm run dev      # hot reload on :5173
npm run build
cp -r dist/* ../src/sbomguard/static/ui/     # ← don't forget this
```

## 5. Architecture

```
data/generator/     rebuilds the synthetic dataset (deterministic, seeded)
data/official/      the REAL SG sample data (6 files)
data/sample_data/   the synthetic reconstruction (5 files)
        │
        ▼
    ingest.py       native CSV · CycloneDX · SPDX   (format sniffed from CONTENT, not extension)
    official.py     adapter for the SG schema + THE DATA-QUALITY CONTROL
        │
        ▼
    graph.py        networkx DiGraph. Transitive paths, diamonds, blast radius.
        │           Depth is RECOMPUTED here, never trusted from the SBOM column.
        ▼
   detectors.py     vulnerability · licence · maintenance   (+ suppression logic)
        │
        ▼
    scoring.py      4 channels → risk_score (anchored) + priority_score (contextual)
        │
        ▼
   analyzer.py      ONE AnalysisResult. One code path computes risk, so the dashboard
        │           and the evaluator can never disagree.
        ├──────────► intel/     classifier · clustering · narrative · osv_client
        ├──────────► features/  remediation · correlation · compliance · feedback · policy_gate
        ├──────────► api.py     FastAPI + the React dashboard
        └──────────► eval/      the proof
```

### The three ideas that carry the project

**1. It is a graph, not a list.** Log4Shell sits three levels down (`App → spring-boot-starter-web
→ …-logging → log4j-core`). Grep a `pom.xml` for "log4j" and you get zero results. A flat scan
finds nothing; graph reachability finds it every time — and reachability is *decidable*, which is
why "100% transitive resolution" is a guarantee rather than an aspiration.

**2. Two scores, because there are two questions.**

| | question | comparable to |
|---|---|---|
| `risk_score` | *How bad is this flaw?* | CVSS, and the ground-truth labels |
| `priority_score` | *What do I fix first?* | nothing else — it is about **us** |

CVSS is deliberately context-free. It doesn't know if the vulnerable function is ever called,
whether the app faces the internet, or whether it touches payment data. **Sorting by CVSS sends an
engineer to patch an unreachable 9.8 in a dev tool while a weaponised 7.5 sits in the payments
path.** On the synthetic data, the *same* Log4Shell CVE correctly ranks 99.3 → 96.4 across four
apps purely on context.

**3. The hard part is NOT detecting things.** A scanner that flags everything scores 100% recall
and is worthless — it gets switched off in week two. Every detector carries explicit,
defensible **suppression**:

- version inside a CVE range but the **build carries a backported fix** → *not vulnerable* (Debian
  and Red Hat do this constantly). We defuse 19/19 planted traps.
- **GPL in an app that is never distributed** → *not a violation*. Copyleft attaches on
  *distribution*. A naive matrix screams "GPL VIOLATION" and is simply wrong.
- **AGPL in the same app IS a violation** — it triggers on *network use*. That's the trap that
  catches teams reasoning by analogy from the GPL.
- **LGPL, dynamically linked, unmodified** → *not a violation*. That's the entire point of the
  "Lesser" GPL.

Suppressed findings are **never deleted** — they're reported separately with their justification.
There is a real difference between a tool that lets you *suppress* risk and one that lets you
*hide* it.

## 6. Non-obvious things that will bite you

| gotcha | why |
|---|---|
| **SG's CSVs are cp1252, not UTF-8** | A plain `open()` dies on the em-dash in the explanation column. `official._read_csv()` handles it. A loader that can't read the file scores 0. |
| **`analyze()` defaults to the OFFICIAL dataset** | Tests and `docs/generate_sample_report.py` explicitly pass `dataset="synthetic"`. If you add a test that calls bare `analyze()`, it will silently run on the wrong data. |
| **The official data has NO Log4Shell** | The UI's headline CVEs are *computed* from the loaded estate (`api._featured()`), not hardcoded. Don't reintroduce a hardcoded CVE — the demo would open on an empty result. |
| **`severity_strategy` must stay `"worst"` in the product** | `"bayes"` scores better on their metric but tells a user a CVSS 9.7 is a MEDIUM. It's confined to the evaluator. See §2. |
| **Depth is recomputed from the graph** | SBOM `depth` columns are written by build plugins and are frequently wrong. Never trust the column. |
| **Version comparison** | `"2.9.9" > "2.9.10"` as strings. Whenever a minor crosses one digit to two, string compare *inverts*. `versions.py` exists for this; there's a test pinning it. jackson-databind's fix for CVE-2019-12384 shipped in exactly that version. |
| **The UI bundle must be rebuilt AND copied** | `npm run build` alone changes nothing that FastAPI serves. |

## 7. Free by design — no keys, anywhere

| capability | service | cost |
|---|---|---|
| Live CVE enrichment | **OSV.dev** | Free. **No API key at all** — not even a signup. |
| LLM narratives | Groq / Google Gemini | Free tier, no credit card. **Optional.** |
| **Narratives with no key** | built-in template engine | **Works offline. This is the default.** |

The deterministic narrative engine composes the analyst write-up from the structured evidence we
already computed. It is genuinely good and needs nothing. The LLM is an *enrichment layer*; if the
key is missing or the call fails, it falls back silently. **Never make the demo depend on a live
LLM call.**

## 9. If you're asked to change something

- **"Make the score higher"** → read §2. Two criteria are provably unreachable and one is
  reachable only by under-reporting severity. Don't chase them. The refusal is the point.
- **"Improve the UI"** → source in `frontend/`, rebuild, **and copy `dist/` into
  `src/sbomguard/static/ui/`**. The Graph view is a hand-drawn SVG orbital layout (no charting or
  graph library) — the app sits at the centre, ring one is code someone chose, every ring beyond
  arrived uninvited, and danger is sorted into a "hot arc" so it clumps instead of sprinkling.
- **"Add a feature"** → the `Finding` object in `analyzer.py` is the atom of the system. Add to
  `features/`, expose via `api.py`, render in `frontend/src/views/`.
- **Before you overwrite anything, commit.** This project was once rebuilt over an
  uncommitted UI and it was unrecoverable. Don't repeat that.
