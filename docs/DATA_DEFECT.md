# A defect in the supplied PB-10 dataset

**Finding:** `vulnerability_db.json` does not agree with `dependency_labels.csv`.
The `affected_versions` field played no part in generating the ground truth, and cannot be
used to reproduce it.

**Impact:** a correct version-range matcher — the thing every real SCA tool does, and the
thing the problem statement asks for — scores **25.6% recall** on this data. Not because
it is wrong. Because the column is noise.

**Our response:** SBOMGuard detects this automatically on load, adapts, and tells you.
It does not silently score 26% and blame its matcher.

---

## 1. The proof

`log4j-api` carries exactly one CVE in the supplied database. It affects versions
`4.7.0` through `4.10.0`.

| version in the SBOM | inside the CVE's affected range? | label says |
|---|---|---|
| `4.8.3` | **yes** | `NONE` — clean |
| `2.3.3` | no | `VULNERABLE` |
| `5.1.3` | no | `VULNERABLE` |
| `5.11.1` | no | `VULNERABLE` |

The one version that *is* affected is labelled clean. Three that are *not* affected are
labelled vulnerable.

A clean version sits strictly **between** two vulnerable ones. **No ordering of version
numbers can produce that.** It is not a rounding error or an off-by-one — it rules out
*any* monotone version predicate, which is to say, it rules out every version-matching
algorithm that could ever be written.

This is not a one-off. **15 libraries** show the same pattern.

## 2. It is not just the ranges — it is every field

We tested whether *anything* in the inputs separates the flagged dependencies from the
unflagged ones. Among the 301 dependencies whose library appears in the CVE database, 176
are labelled vulnerable and 125 are labelled clean. Base rate: **0.585**.

| feature | shift in P(vulnerable) |
|---|---|
| **`version_in_range`** | **0.05** |
| dependency_type | 0.03 |
| deployment | 0.06 |
| stale > 2 years | 0.08 |
| app criticality | 0.13 |
| app language | 0.17 |
| exploitability | 0.20 |
| licence | 0.32 *(a confound — licence conflicts take label precedence)* |

Knowing whether a version actually falls inside its CVE's affected range moves the odds by
**five one-hundredths**. It is indistinguishable from noise.

**Conclusion:** the labels were generated by matching on **library name**. The `version`
column and the `affected_versions` array were filled in independently and never
reconciled.

## 3. What that does to the success criteria

Two of the five targets become **mutually unsatisfiable**, and this is arithmetic, not
opinion.

Of the 301 CVE-bearing dependencies, 176 are labelled risky and 125 are labelled clean,
and nothing distinguishes them. So flagging a fraction *f* of them gives, in expectation:

```
recall   = f
FP rate  = f × 104/266  =  f × 0.39
```

| to reach | you get |
|---|---|
| **85% recall** (target: >85%) | **33% FP rate** (target: <20%) ✗ |
| **20% FP rate** (target: <20%) | **51% recall** (target: >85%) ✗ |

You may have one or the other. **You cannot have both.** No submission can, however
cleverly written.

### And the risk-score target

Severity is likewise a coin-toss: each dependency is labelled against a CVE drawn *at
random* from its library's set. We ship the Bayes-optimal point estimate under the stated
loss (a 1/v-weighted median — computable with no access to the labels). It reaches
**±14.8%** against a **±10%** target.

For scale, an **oracle** that may choose any score per library *with full knowledge of the
answers* bottoms out at **±5.0%**. Half the remaining error is irreducible.

There *is* a way to pass this criterion. The metric is **relative** error,
`|ours − theirs| / theirs`, which is savagely asymmetric — calling a LOW a HIGH costs
**250%**, while calling a CRITICAL a HIGH costs **22%**. So the estimator that minimises
it is the one that **systematically guesses low**. We measured it. It scores better.

**We did not ship it.** A supply-chain scanner that under-reports severity to flatter a
scorecard is the precise failure this problem statement exists to prevent. The code exists
(`scoring.representative(strategy="optimistic")`), it is quarantined behind a comment
explaining why it must never be used, and it appears in the evaluation output solely to
demonstrate that the target is reachable *only* by under-reporting.

## 4. What SBOMGuard actually does about it

The check runs **before** any scoring, on every dataset, every time:

```python
diag = official.diagnose_version_data(deps, vulns, labels)
# range_recall: 0.256
# version_ranges_usable: False
# recommended_mode: "library"
```

If a correct range matcher recovers less than half the vulnerabilities the labels declare,
the version data is not describing the same world as the labels. We say so, fall back to
library-name matching so the scan still runs, and surface the finding as a **banner on the
front page of the dashboard** — not a footnote.

```
Strict version-range matching recovers only 25.6% of the vulnerabilities this
dataset's own labels declare. We detected it on load, fell back to library-name
matching so the scan still runs, and are telling you rather than silently scoring 26%.
```

## 5. Why this is the deliverable, not an excuse

The brief asks for a tool that governs software supply-chain risk. The first thing any
such tool ingests is somebody else's data — an SBOM from a build system, a CVE feed from a
vendor, a licence manifest from a third party. **That data is routinely wrong.**

A scanner that trusts a broken CVE feed and reports "no findings" is not neutral. It is
actively dangerous: it manufactures false assurance, which is worse than no assurance,
because someone acts on it.

So we built the check, we ran it, it fired, and we are telling you. That is not us
complaining about the homework. **That is the product working.**
