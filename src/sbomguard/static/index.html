A critical CVE just dropped. Which of our applications are affected?

In December 2021 this question took most organisations four days to answer — not because Log4Shell was hard to understand, but because nobody knew what was actually in their software. Here it is answered by graph reachability, in milliseconds, including through transitive dependencies that appear in no manifest anyone has ever read.

The priority queue

Two numbers, deliberately. Flaw is how bad the vulnerability is in the abstract — comparable to CVSS. Priority is how bad it is for us: is the vulnerable function even reachable, is there an exploit in the wild, is the app internet-facing, does it touch cardholder data. Sorting by CVSS sends an engineer to patch an unreachable 9.8 in a dev tool while a weaponised 7.5 sits in the payments path.

Remediation playbook

Risk archetypes

265 findings are not 265 problems. They are a handful of recurring patterns — and each one is a single campaign with a single owner.

ML early-warning

The classifier never sees the CVE table. It learns the profile of a component that gets breached — old, unmaintained, buried deep, sitting in an exposed app. Where the rules see nothing and the model sees danger, you have a Log4j in November 2021.

Cross-application leverage

Fix by leverage, not by application. Patching the same library ten times in ten repos is ten times the work for the same outcome.

Compliance posture

Every control carries its evidence, not just a verdict — which is what an auditor actually asks for.

CI/CD policy gate

Everything else here reports on risk. Reports do not stop risk being merged. This returns a UNIX exit code, so it can fail a build. Log4Shell entered those codebases not because nobody had a scanner, but because nothing in the pipeline had the authority to say no.
